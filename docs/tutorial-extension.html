VS Code walkthrough

The same iteration as the CLI and Claude Code tutorials, run through the VS Code extension. You click the board, drag cards, and fill inline forms; the extension shells out to the am CLI under the hood, one verb per action.

What this covers

The full lifecycle inside the Agile Markdown VS Code extension: a board with Current and Icebox columns, a Plan tab for IPM, a detail panel with the state machine and acceptance ceremony, drag-and-drop reorder, and an inline reject form. Same project as the other two tutorials: a command palette for a notes app.

If you have not installed VS Code, Claude Code, the am binary, or run am init in your code repo yet, see Before you start on the tutorials index. Open the panel with the command palette: Agile Markdown: Open Board.

Inception

The extension does not have a built-in inception editor; it shares inception.md with the rest of the surfaces. Open the file in VS Code and write the six sections.

Action Command palette → Agile Markdown: Open Item → pick inception.md. Or open it from the file tree. Edit the six headed sections (user, goal, reason, success, constraints, out of scope). Save.
Coach skill If you also have Claude Code attached to the same workspace, /am-inception walks you through the questions. The extension does not run skills directly; it shares state with whatever agent is in the workspace.

Backlog and stories

The extension's Add story button (the + in the Current column header) creates an item. The story lands in the icebox so the rank is yours to set.

Action Click + in the Current column. The card flips to an inline title input. Type "Open palette on cmd-k", press Enter. Repeat for the other two stories. Each appears in the Icebox column as you create them.

Click each card to open the detail panel. The new Acceptance section sits above Description; replace the placeholder bullets with concrete criteria. Save the description (the textarea + Save action below) when you are done editing.

Action Click the points pip for each story (1, 2, 3, 5, 8). The card's points badge updates immediately.

Iteration planning

Drag the three stories from Icebox into Current. Within Current, drag to reorder. The top of Current is what the dev will pull next.

Action Click the Plan tab in the topbar. The view loads the iteration plan: stories committed up to rolling velocity, plus a below-line backlog and a warnings list (missing acceptance, oversized features, unestimated features, overcommit).

Plan tab · committed

#  Title                            Type     Pts  Acceptance  Status      Flags
1  Open palette on cmd-k            feature   3   3 bullets    unstarted
2  Fuzzy match note titles          feature   3   3 bullets    unstarted
3  Recent searches dropdown         feature   3   3 bullets    unstarted

Committed: 9 pts. Velocity: 10. Fits.

If a story is missing its ## Acceptance section the Plan tab flags it. Click into the card and add bullets before pulling.

The iteration

Story 1 · clean accept

Click the top card. The detail panel opens with the acceptance criteria visible. The dev writes the code; you advance the state machine.

Action Click started on the state machine. Card status pill flips. After the code lands, click finished, then delivered. Each click writes the corresponding timestamp to the body's frontmatter.

Click accepted. The button does not flip the status. An inline PM ceremony panel appears with the criteria and a Yes / Cancel pair. Read the criteria; verify the work; click Yes.

Inline PM ceremony

As PM, do you accept?
Story         Open palette on cmd-k
Type          feature
Estimate      3 pts
Status        delivered -> accepted

What to verify
[~] cmd-k opens the palette overlay
    claim: opens, focused, mouse position preserved
[~] the palette captures keyboard focus on open
[~] escape closes it

[ Yes, accept ]   [ Cancel ]

The extension renders each acceptance bullet as a checkbox. The dev pair flips bullets to [~] claimed at delivery time; the PM ticks each one to verify or unticks to reject. When every bullet is verified, the Accept button is enabled. The points roll into velocity for the iteration.

Story 2 · refused estimate, decompose

Open the Fuzzy match card. Mid-build, the dev decides this is bigger than 3. Click the 5 pip. Then click 8. Then 13. The 13 pip is greyed out; the cap is 8.

Try anyway via the CLI to see the refusal text:

Coach refusal

Refused: features over 8 points are epics.
Source: 8-point-hard-cap
Next: split this story; keep each piece at or below 8 points.

Open the file directly, split into "Search index" (5) and "Search typo tolerance" (3, into icebox). Or, in another VS Code window, run /am-decompose in Claude Code and let the skill draft the split. Either way the original story closes; the new one slots into the iteration.

Story 3 · rejection with reason

The Recent searches story ships, you click delivered. Acceptance ceremony renders. You spot the bug: the dropdown shows stale entries from the last session. Click Cancel, then click the rejected button.

An inline reason form appears.

Reject with reason

Reject: record the reason

[ stale entries from the prior session leak into the dropdown ]

[ Reject with reason ]   [ Cancel ]

Submit. The body picks up a dated ## Rejection notes entry. Status flips to rejected. The dev re-pulls (status auto-restarts), fixes the bug, marks delivered. The acceptance ceremony renders again. Yes.

Retro

The extension does not have a Retro tab today. Open a terminal in VS Code (Ctrl/⌘ + `), run am retro, and capture outputs from there.

Terminal · retro outputs

$ am retro
Retro summary

  velocity:        9 pts
  volatility:      0%
  cycle time:      0.4d (median)
  rejection rate:  0% (latest, target band 5-15%)
  accepted total:  3
  rejected total:  0

Three questions:
  1. What worked?
  2. What did not work?
  3. What changes for next iteration?

$ am record-learning "Plan tab caught two missing acceptance sections before pulling. Worth keeping the warning loud."
$ am team-agreements --add "Recent-searches list reads from disk on focus; never from in-memory cache."
Bootstrap state First-iteration retro numbers are not meaningful yet. Velocity reads bootstrap until the rolling lookback fills in. Rejection rate buckets only completed iterations. The numbers stabilize from iteration three; before that, retro is conversation, not metrics.

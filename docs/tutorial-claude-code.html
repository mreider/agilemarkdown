What this covers

Claude Code in VS Code, with agilemarkdown installed. The coach stance lives at .claude/agilemarkdown-coach.md and is pulled into CLAUDE.md via @-import. Six skills (am-inception, am-plan, am-align, am-decompose, am-accept, am-retro) auto-fire on the right phrases. The hook at .claude/hooks/coach-gate.sh blocks any tool call that would break a hard rule.

Setup steps live in Before you start on the tutorials index. After am init in your code repo, open Claude Code and start chatting.

Inception

You
"Let's run an inception. I want to build a command palette for my notes app."
Claude
→ /am-inception auto-fires. Skill walks the six questions one at a time.

"Who, specifically, are we building this for? Not a demographic, a real person with real circumstances."
You
"Me. I keep notes in a markdown vault, 1200+ files, switch windows constantly, and the mouse is too slow."

Claude asks the remaining five questions one at a time. After all six answers, the skill calls inception_doc with the assembled body and writes inception.md. Claude prints the doc back and asks if you want to commit it.

Backlog and stories

You
"Make a backlog and seed three stories: cmd-k opens a palette, fuzzy match note titles, recent searches dropdown. Each is a feature, points around 3."
Claude
→ create_backlog(name="notes")
→ create_item × 3
→ set_estimate × 3 (3 pts each)
→ set_description × 3 with ## Acceptance bullets per story (drafted from the inception)

"Three stories created with acceptance criteria. Want me to walk you through each before we plan the iteration?"

You read each one. Edit a bullet here, add one there. The agent uses set_description to land the changes. The bullets are concrete; you can check them at acceptance time.

The iteration

Story 1 · clean accept

You
"Yes."
Claude
→ /am-align auto-fires. The skill reads the story, restates the agent's understanding in one paragraph, lists the acceptance bullets verbatim, asks "anything to clarify before I pull?"
You
"Looks right. Pull it."
Claude
→ coach_check(action="pull", path=...) returns allowed. set_status(path, "started") goes through.
Writes the code. Tests pass.
→ set_status(path, "finished")
→ For each bullet the agent believes it satisfied: set_acceptance_state(path, index=N, state="claimed", claim_note="...").
→ set_status(path, "delivered")

→ /am-accept auto-fires.
→ acceptance_prompt(path)
Claude
Story: Open palette on cmd-k (notes/Open-palette-on-cmd-k.md)
Type: feature
Status staging: delivered -> accepted
Estimate: 3 pts
What to verify:
  [~] cmd-k opens the palette overlay
       (claim: opens, focused, mouse position preserved)
  [~] the palette captures keyboard focus on open
  [~] escape closes it

As PM, do you accept?
Claude
"Bullet 1: does cmd-k open the palette? Yes or no?"
You
Verify the bullet manually. Yes.
Claude
→ set_acceptance_state(path, index=1, state="verified"). Walks bullets 2 and 3 the same way; both pass.
→ set_status(path, "accepted")
Hook gates this. Verdict: allowed, because the ceremony rendered and every bullet is verified. The status flips. Velocity ledger gains 3 points.

Story 2 · 8-point cap refusal

Claude
Mid-build, the agent decides this is bigger than first thought.
→ coach_check(action="set_estimate", path=..., estimate=13, type="feature")
Coach
Refused: features over 8 points are epics.
Source: 8-point-hard-cap
Next: split this story; keep each piece at or below 8 points.
Claude
"Cap is 8. I want to split this into Search index (5 pts) and Search typo tolerance (3 pts). Index ships this iteration; typo tolerance into icebox. Sound right?"
You
"Yes."
Claude
→ /am-decompose auto-fires.
→ create_item, set_estimate, set_description with criteria, for each new story.
The original closes. Search index pulls in. Build, finish, deliver, ceremony, accept.

Story 3 · rejection with reason

Recent searches dropdown ships. The agent renders the ceremony with all three bullets claimed.

Claude
"Bullet 2: do recent searches show only the current session? Yes or no?"
You
"No. The dropdown shows stale entries from the prior session. Reject bullet 2 with that reason."
Claude
→ reject_item(path, reason="stale entries from prior session leak into dropdown", failing_bullet=2)
Bullet 2 reopens from [~] to [ ]. Body picks up a dated ## Rejection notes entry citing the bullet. Status flips to rejected.

"Got it. Pulling again to fix bullet 2."

The agent re-starts, fixes the bug, re-claims bullet 2 with a new note, delivers. The ceremony renders again. You verify. Accept.

Retro

You
"Run the retro."
Claude
→ /am-retro auto-fires.
Pulls dashboard(), velocity_history, type_mix, reads learnings.md and team-agreements.md. Renders the summary, asks the three questions one at a time.

"What worked?"
You
"Splitting the 13-pointer mid-iteration, fast. Coach refusal saved a guess."
Claude
→ record_learning("Split a 13-pointer in 10 min. Coach refusal was the trigger.")

"What did not work?"
You
"Recent searches store leaked across sessions. Worth an agreement: state lives in disk, not memory."
Claude
→ team_agreements(add="Cross-session UI state reads from disk on focus, never from in-memory cache.")

"What changes for next iteration?"

You answer the third question; the agent records it. The retro updates learnings.md and team-agreements.md, then closes the iteration.

Bootstrap state The numbers in the retro summary stabilize from iteration three. Velocity, volatility, cycle time, and rejection rate are not meaningful yet on a fresh project.

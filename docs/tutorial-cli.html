What this covers

A small product team adopts agilemarkdown for an internal tool. They run one full iteration and start the second. Every command in this tutorial runs against the fixture from tests/fixtures/build-sample.sh, so you can replicate it locally.

The lifecycle is four phases. Inception is rare (once per project). Iteration planning happens at the start of every iteration. The iteration itself is where stories flow through states, and PM acceptance happens continuously, not in a single demo at the end. Retro closes the iteration.

This is the XP cadence, not the Scrum cadence. The iteration plan is a forecast, not a commitment; the team finishes what it finishes. Acceptance is continuous; there is no end-of-sprint demo. Retro is a Friday conversation, not a status report.

If you have not installed agilemarkdown yet, see Before you start on the tutorials index. By the end of those steps your repo has the coach files installed and the MCP server is wired into Claude Code.

Inception

The team is starting a new internal tool. They could just start writing stories, but they choose to spend twenty minutes on an inception first. The output is a one-page document.

Run the inception verb

shell · bootstrap
$ cd ~/code/internal-tool
$ am init                       # installs the coach files
$ am inception                   # seeds inception.md
wrote inception.md (template)
Edit the six sections (user, goal, reason, success, constraints, out of scope) before pulling stories.

The team opens inception.md and works through the six sections together. The PM types; the dev pair (or Claude Code) asks the questions. After twenty minutes they commit the document.

Coach skill Running /am-inception in Claude Code triggers the bundled skill. The agent walks the human through the six questions one at a time, captures the answers, and writes inception.md via the inception_doc MCP tool. The human edits before saving.

What lands on disk

inception.mdOne page. Six headed sections: user, goal, reason, success, constraints, out of scope.

That is the whole inception artifact. No backlog yet. No stories. Stories come from the inception, but they get written once the team is ready to plan the first iteration.

Backlog and stories

The team has the inception. Now they need a backlog folder and a few stories. This is the bridge between the one-page narrative and the rank-ordered priority list.

Create the backlog folder

shell · backlog
$ am create-backlog product

This creates product/ alongside the code, plus a product.md overview. The same step seeds three sample stories (a feature, a bug, a chore). They are intentional onboarding examples; if you do not want them in your project, delete them: rm product/Sample-*.md.

Add stories from the inception

shell · stories
$ cd product
$ am create-item "Login flow"
$ am create-item "Search relevance"
$ am create-item "Saved searches"

Each call writes a .md file with the standard frontmatter (type: feature, status: unstarted) and a body with sections for Problem statement, Possible solution, Acceptance, Comments, and Attachments. The ## Acceptance section starts with placeholder bullets.

Write real acceptance criteria

Open each story and replace the placeholder bullets with concrete criteria. Two to four bullets per feature. Each is something the PM can check; not a property of the code. Bullets ship as GitHub-style checkboxes: [ ] open by default, [~] when the dev pair claims the bullet is satisfied, [x] after the PM verifies it at acceptance time.

product/Login-flow.md
## Acceptance

- [ ] a registered user can sign in with email + password
- [ ] a wrong password produces a recoverable error, not a 500
- [ ] the session persists across a page reload

Add a bullet from the shell with am acceptance add product/Login-flow.md "..."; mark one claimed with am acceptance claim product/Login-flow.md 1 --note "..."; mark verified with am acceptance verify .... The CLI is one path; the agent skills do the same writes on your behalf. Without this section in the body the agent has no target, and coach_check action=pull refuses. Writing the criteria up front is what keeps the PM ceremony from falling back to "review the diff".

Estimate

Estimating in Pivotal is about uncertainty, not effort. The Fibonacci scale (1, 2, 3, 5, 8) maps to confidence: 1 is well-trodden, 8 is the hard cap above which the story is an epic. am estimate --advise prints the framing in the terminal if you need a refresher.

"A 2-point story for design should equate to the same amount of effort as a 2-point story for a developer. Don't estimate using time. If we find a story is above 3 points, or requires a huge amount of effort, this is a red flag that the story should be broken down."

Pivotal Tracker help center · estimating stories

shell · estimate
$ am estimate Login-flow.md 5
$ am estimate Search-relevance.md 3
$ am estimate Saved-searches.md 5

Sync

Run am sync from the project root. Sync regenerates index.md, velocity.md, and friends; validates the schema; and lands every active item in _icebox.md in the order they were created. Items will move from icebox to priority during the next phase.

Iteration planning

The team has dropped a dozen story ideas into _icebox.md over the last week. Now it is Monday morning, time to plan the first iteration. The PM has been ranking the icebox in the days leading up to today, so the top of icebox is more or less in the order the team will work it.

Promote stories from icebox to priority

shell · planning prep
$ cd product
$ am unice Login-flow.md --top
$ am unice Search-relevance.md
$ am unice Dark-mode.md
$ am unice Saved-searches.md
$ am unice Pagination.md
$ am unice API-tokens.md

Six features promoted into the priority list. The PM also moves a bug ("Login bug") and a chore ("Rotate API key") in.

Render the plan

shell · sprint plan
$ am sprint plan
Iteration plan (product)   velocity 8 / iteration

   1. ★  Login flow                       feature 5p   acceptance: 3 bullets
   2. ★  Search relevance                 feature 3p   acceptance: 3 bullets

Backlog (below the line, not committed):
   3. ★  Dark mode                        feature 8p   acceptance: 3 bullets
   4. ★  Saved searches                   feature 5p   acceptance: 3 bullets
   5. ★  Pagination                       feature 3p   acceptance: 3 bullets
   6. ★  API tokens                       feature 5p   acceptance: 3 bullets
   7. ●  Login bug                        bug      -    acceptance: 0 bullets
   8. ○  Rotate API key                   chore    -    acceptance: 0 bullets

Committed: 8 pts across 2 stories. Velocity: 8.

Velocity is 8 (rolling, computed from the last few iterations). Login flow and Search relevance fit. Dark mode is right there at the top of the below-the-line set; the team flags that it might creep in if Login flow finishes ahead of schedule. The bug and chore stay below the line; the team will pull them when they have slack.

"Tracker auto-plans the future iterations of the Backlog by stuffing them with stories from the top down until each iteration's accepted points reach the project velocity. The current iteration is filled the same way and then shifts to whatever the team has actually started."

Pivotal Tracker help center · planning with velocity

The commit moment There is no separate "commit" file. The act of leaving the rank order in _priority.md as-is and starting to pull stories is the commit. Pivotal Tracker worked this way too.

The iteration

The dev pair (Claude Code) and the human (the PM, who is also the developer in solo mode) work through the priority list. The agent pulls the next unstarted story, writes the code, marks the story finished and then delivered. The human reviews, runs the acceptance criteria, says yes or no.

Story 1 · Login flow

You
"What's next?"
Claude
→ next_item() returns product/Login-flow.md. The story has acceptance criteria; estimate 5; not blocked. Pulls.
Claude
→ set_status(Login-flow.md, "started")
Writes the code. Auth middleware, session cookie, /forgot-password route. Tests pass. Marks finished, then delivered.
Claude
→ acceptance_prompt(Login-flow.md): 
Story: Login flow (product/Login-flow.md)
Type: feature
Status staging: delivered -> accepted
Estimate: 5 pts
What to verify:
  [~] a registered user can sign in with email + password
       (claim: passes tests/auth_test.go)
  [~] a wrong password produces a recoverable error
  [~] the session persists across a page reload

As PM, do you accept?
You
Tries each bullet. They pass. "Yes, accept."
Claude
→ set_status(Login-flow.md, "accepted")
Velocity ledger gains 5 pts.

Story 2 · Search relevance

The agent pulls Search relevance. Halfway through, the agent realizes it is bigger than 3 points. It re-estimates and tries to set the new value:

Claude
→ coach_check(set_estimate, path=Search-relevance.md, estimate=13)
Coach
Refused: features over 8 points are epics.
Source: 8-point-hard-cap
Next: split this story; keep each piece at or below 8 points.
Claude
"I'll split it. The first half is the search index; the second is the typo tolerance. Want me to call am-decompose?"
You
"Yes. Index first; typo tolerance can wait until next iteration."
Claude
Splits into two stories: Search-index.md (5 pts) and Search-typo-tolerance.md (3 pts, into icebox). Closes the original. Pulls Search-index.md.

The dev pair finishes the index, marks delivered. The PM accepts. Velocity ledger gains another 5 points (now 10 for the iteration; the team is over the 8 forecast, which is fine, because yesterday's weather is a forecast, not a budget).

A rejected story

The team pulls Dark mode (the next-ranked feature). The dev pair builds it. The PM looks at the result and notices the system-prefers-dark default is wrong: it picked up the OS setting at component mount, not at session start, so a user who toggled their OS theme during the session got jumbled colors.

You
"No. The system-prefers-dark is supposed to be the default for new users only, set at signup. This is mid-session theming. Reject."
Claude
→ reject_item(Dark-mode.md, reason="system-prefers-dark applies at signup, not mid-session")

The story is back in the iteration with status rejected. The agent re-pulls it, fixes the bug, marks delivered again. The PM re-accepts.

Rejection is a ritual Pivotal counted rejection as data, not as failure. A healthy rejection rate is 5–15%. Below that the PM is rubber-stamping; above it the team is shipping the wrong thing. The retro reads the latest rejection_rate and asks why.

Retro

End of the iteration. The team runs am retro and looks at the numbers before the conversation.

shell · retro summary
$ am retro
Retro summary

  velocity:        10 pts
  volatility:      28%
  cycle time:      1.6d (median)
  rejection rate:  10% (latest, target band 5-15%)
  accepted total:  3
  rejected total:  1

Three questions:
  1. What worked?
  2. What did not work?
  3. What changes for next iteration?

Record outputs:
  am team-agreements --set "..."      (new agreement)
  am record-learning "..."            (one-line learning)

Velocity hit 10 (8 forecast plus the bonus from finishing early). Rejection rate is 10%, in band. Cycle time of 1.6 days median feels right.

Bootstrap state On a brand-new project the retro numbers are not meaningful yet. Velocity reads as (bootstrap) until the rolling lookback window has real iterations behind it. Rejection rate buckets only completed iterations, so a story rejected today does not show in the rate until the iteration closes. Cycle time medians on a single short iteration are likewise misleading. The numbers stabilize from iteration three onward; before that the retro is conversation, not metrics.

The team answers the three questions. The PM types:

shell · retro outputs
$ am record-learning "Splitting the 13-pointer cost ten minutes and saved an iteration of confusion."
$ am record-learning "Mid-session theming is a real edge. Add it to the dark-mode acceptance bullets next time."
$ am team-agreements --add "Stories that touch theming check OS prefs at session start, not at component mount."

Use --add to append a new agreement bullet to the existing list. --set overwrites the file and is for bootstrapping or resetting the agreements wholesale.

That is the whole retro. Two learnings, one new working agreement. The agreement gets read at the start of the next session by every agent that opens this repo, because team-agreements.md is part of the coach's onboarding.

A second iteration

Monday morning again. The PM ranks the icebox; the team runs am sprint plan; the agent pulls the first story. The shape is identical. The difference is rolling velocity now reflects iteration one (10 points), so the iteration-two budget is 10. The team commits to two features and the bug they queued from week one.

Halfway through the iteration, the dev pair starts a story that has no ## Acceptance section in its body. The coach surfaces a nudge:

Coach
Nudge: feature has no acceptance criteria.
Source: acceptance-criteria
Next: add a `## Acceptance` section with bullets to the body before
starting, or skip with intent.

The PM stops the agent, drafts three bullets into the body, says go. Agent pulls.

By the end of iteration two the team has shipped two features and a bug. They run the retro. Velocity is now 11 (rolling). The rolling forecast has nudged up. Yesterday's weather: tomorrow looks like today.

Scripting against the CLI

Every read view above has a --json sibling, and there are six data-export verbs designed for piping. The shapes are the same Go structs the MCP server returns, so anything you script against the CLI will keep working when you switch to the MCP path, and vice versa.

shell · piping the JSON surface
$ am next | awk 'NR==1{print $1}' | xargs am get-item | jq .title
# "Search relevance"

$ am list-items product --status started | jq -r '.items[].title'
# Work in flight, one title per line.

$ am dashboard --json | jq '{velocity, accepted: .stories_accepted_total}'
# A two-field summary you can drop into a status bar or shell prompt.

$ am list-backlogs | jq -r '.backlogs[]' \
    | while read b; do am list-items "$b" \
        | jq -r --arg b "$b" '.items[] | select(.blocked) | "\($b)\t\(.title)"'; \
      done
# Every blocked story, across every backlog.

The same surface is what the VS Code extension uses; the extension shells out per click instead of hosting a long-lived MCP child. Anything the board does, your shell can do.
